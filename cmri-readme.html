Architecture README

CMRI integration overview

This document explains what the CMRI-related pieces in this repo do, how they interact, and why they were split into separate components instead of putting everything into one class.

Current implementation uses CmriSubroutines directly and supports Memory, Tcp, and Serial transport modes.

Big picture

The server has one job: connect to a CMRI node, poll it for input bytes, and publish the latest values to connected web clients over SignalR. 

Program.cs
    -> configures DI and binds CMRI settings
    -> creates CmriService as a singleton
    -> starts DevicePollingService as a hosted background worker

DevicePollingService
    -> repeatedly calls CmriService.ReadAsync()
    -> pushes the returned byte[] to SignalR clients via CmriHub

CmriService
    -> chooses the correct transport (Memory / Tcp / Serial)
    -> creates CmriSubroutines.Subroutines
    -> initializes the configured node
    -> reads CMRI input bytes from the node

Transport
    -> defines how bytes physically move
    -> Memory = local fake device
    -> Tcp = ser2net / network connection
    -> Serial = direct COM port

Components and responsibilities

Server/Program.cs

Responsibility: application composition.

  • Binds the Cmri configuration section into CmriOptions.
  • Registers CmriService in dependency injection.
  • Registers DevicePollingService as a hosted service.
  • Maps the SignalR hub at /cmriHub.

It decides what gets created, not how CMRI works.

Server/Services/CmriOptions.cs

Responsibility: configuration shape.

  • Transport selection: Memory, Tcp, Serial.
  • Node settings: NodeAddress, NodeType, optional Ct.
  • Connection settings: host, port, COM port, baud, buffer sizes.
  • Mock values: MockInputs for memory mode.

This is data only. No behavior.

Server/Services/CmriService.cs

Responsibility: CMRI application service.

  • Reads options from DI.
  • Chooses the transport implementation.
  • Creates Subroutines from the external CmriSubroutines library.
  • Initializes the target node.
  • Exposes ReadAsync for the rest of the app.

This is the boundary between your app and the CMRI library.

Server/Services/CmriMemoryLoopbackTransport.cs

Responsibility: test-friendly in-memory device simulation.

  • Wraps MemoryTransport from CmriSubroutines.
  • Observes outgoing poll messages.
  • When it sees a poll request, it enqueues a valid CMRI response frame.
  • Returns configured input bytes without requiring real hardware.

This is not business logic. It is infrastructure for local simulation.

Server/Services/DevicePollingService.cs

Responsibility: background polling loop.

  • Runs continuously while the server is alive.
  • Calls CmriService.ReadAsync() every 500ms.
  • Sends each result to all SignalR clients.
  • Handles cancellation and logs failures.

It schedules work. It does not know transport details.

Server/Hubs/CmriHub.cs

Responsibility: SignalR endpoint.

  • Represents the real-time connection surface for web clients.
  • Lets the server push ReceiveMessage events to connected browsers.

The hub is the delivery mechanism, not the CMRI reader.

Startup flow

  1. Program.cs loads configuration.
  2. Program.cs binds the Cmri section to CmriOptions.
  3. Dependency injection creates a singleton CmriService.
  4. CmriService chooses a transport based on CmriOptions.Transport.
  5. CmriService creates Subroutines from the external CMRI library.
  6. CmriService calls Init on the configured node.
  7. DevicePollingService starts in the background.
Important: the application does not talk to a serial port or TCP socket directly. It always goes through CmriService, which goes through a transport, which is then consumed by CmriSubroutines.Subroutines.

Runtime polling flow

1. DevicePollingService wakes up
2. DevicePollingService calls CmriService.ReadAsync(token)
3. CmriService calls Subroutines.InputsAsync(nodeAddress, token)
4. Subroutines sends a CMRI poll frame through the configured transport
5. Transport returns bytes from one of these sources:
   - Memory: generated loopback response
   - Tcp: bytes from ser2net / network socket
   - Serial: bytes from COM port
6. Subroutines parses the CMRI response into a byte[]
7. CmriService returns the byte[] to DevicePollingService
8. DevicePollingService pushes the byte[] to SignalR clients using "ReceiveMessage"
Layer Knows about Does not need to know about
DevicePollingService When to read and when to publish TCP framing, COM ports, CMRI initialization details
CmriService CMRI setup and transport selection SignalR client management or UI concerns
Transport How bytes move Polling schedule or web delivery
CmriHub Real-time client connection Where data came from

Why the memory transport wrapper exists

This is the one part that is easy to miss.

MemoryTransport is only a byte pipe. It can hold bytes in memory, but by itself it does not act like a CMRI node. Your polling code sends a CMRI poll frame and expects a correctly shaped CMRI response frame back.

CmriMemoryLoopbackTransport bridges that gap. It watches for a poll request and immediately queues a valid response: 

Poll request sent by Subroutines  ->  wrapper detects 'P' message
                                  ->  wrapper enqueues STX + UA + 'R' + input bytes + ETX
                                  ->  Subroutines reads that response as if it came from real hardware
Result: the app can run end-to-end without a physical device, while still exercising the real Subroutines CMRI parsing path.

Separation of concerns

Concern Current owner Why that is useful
Application startup and DI Program.cs All composition is in one place.
CMRI configuration values CmriOptions Settings are explicit and easy to change per environment.
CMRI protocol usage CmriService The rest of the app avoids direct dependency on low-level setup code.
Hardware / connection details transport classes Switching between Memory, Tcp, and Serial stays localized.
Periodic polling DevicePollingService Read frequency and retry behavior live in one background worker.
Browser delivery CmriHub / SignalR Real-time web delivery is isolated from CMRI logic.
If everything were in one class, that class would need to know: configuration, node initialization, transport setup, polling loop timing, error handling, and SignalR broadcasting. That would make the code harder to change and harder to reason about.

Configuration

The main switch is the Cmri section in appsettings.json.

Memory mode

{
  "Cmri": {
    "Transport": "Memory",
    "NodeAddress": 0,
    "NodeType": "SMINI",
    "MockInputs": [1, 0, 0]
  }
}

TCP mode via ser2net

{
  "Cmri": {
    "Transport": "Tcp",
    "NodeAddress": 0,
    "NodeType": "SMINI",
    "Host": "CmriPi",
    "Port": 3333
  }
}

Serial mode

{
  "Cmri": {
    "Transport": "Serial",
    "NodeAddress": 0,
    "NodeType": "SMINI",
    "ComPort": 5,
    "Baud100": 1152
  }
}

Mental model

  • Program.cs wires the pieces together.
  • CmriOptions says how the CMRI connection should be configured.
  • CmriService is the app-facing CMRI entry point.
  • Transport is the wiring under the service.
  • DevicePollingService is the metronome.
  • CmriHub is the broadcast channel to browsers.
Short version: your app now has a clean pipeline of configuration -> CMRI service -> polling worker -> SignalR hub -> browser.
